# 무한매수법 자동화 웹앱 — Claude Design 핸드오프

> **용도:** 이 문서는 Claude Design이 UI/UX를 설계하기 위한 단일 레퍼런스다.
> 비즈니스 로직·공식·화면 구조·데이터 모델이 모두 담겨 있다.
> 디자인 결과물이 이 문서를 완전히 커버해야 핸드오프 성공이다.

---

## 1. 프로젝트 개요

### 무엇을 만드는가

"라오어 무한매수법 V4.0" 운용을 자동화하는 **다중 사용자 웹 대시보드**다.

무한매수법(무매)은 TQQQ/SOXL (미국 3배 레버리지 ETF)을 매일 LOC(Limit On Close, 종가 근처 지정가) 주문으로
분할 매수·매도하는 장기 전략이다. 사용자는 매일 아침 당일 주문 가격과 수량을 확인하고, 증권사 앱에서
직접 주문을 넣는다. 장 마감 후 체결된 내역을 앱에 입력하면 시스템이 상태값(T·평단·잔금)을 자동 갱신하고
내일 계획을 다시 계산해준다.

### 현재 상황 (자동화 전)

- 상태파일이 `.md` 파일로 로컬에 존재
- 매일 사용자가 체결 결과를 Claude에게 자유서술로 보고 → Claude가 계산·갱신
- 사람이 직접 계산하는 구조라 실수 가능성 존재, 모바일 접근 불가

### 목표

| 지금 | 자동화 후 |
|---|---|
| Claude에게 자유서술 보고 | 웹앱 UI에서 체결 입력 |
| Claude가 수동 계산·MD 갱신 | 서버가 자동 계산·DB 저장 |
| 로컬 파일, 단일 기기 | 클라우드, 다중 기기·다중 사용자 |

### 사용자 페르소나

- **주 사용자**: 라오어 무한매수법을 실제로 운용 중인 개인 투자자
- **기술 수준**: 비개발자. 증권사 앱에 익숙한 수준
- **사용 패턴**: 하루 2회 접속 — 아침 (오늘 주문 확인), 저녁 장 마감 후 (체결 입력)
- **핵심 니즈**: 계산 실수 없이 오늘 LOC 주문 가격·수량을 빠르게 확인

## 3. 핵심 도메인 개념

디자이너가 UI 레이블과 화면 구조를 정확히 잡으려면 아래 용어를 반드시 이해해야 한다.

### T값 (Turn)

진행 회차를 나타내는 소수점 숫자. 정수가 아니며 소수점 제한 없음.

```
1회 매수 체결  → T += 1
절반 매수 체결 → T += 0.5
쿼터매도 체결  → T = T × 0.75
지정가매도 후 LOC 매수 1회 체결  → T = T × 0.25 + 1
지정가매도 후 LOC 매수 절반 체결 → T = T × 0.25 + 0.5
```

### 평단 (평균단가)

보유 주식의 평균 매입가. 매도 시 변동 없음.

```
매수 시: 평단_new = (평단_old × 수량_old + 체결가 × 체결수량) / (수량_old + 체결수량)
```

### 잔금

원금 중 아직 매수에 사용하지 않은 금액. 매수 시 감소, 매도 시 증가.

### 별% (★%)

별지점을 계산하는 기준 퍼센트. T가 커질수록 감소하며, 후반전에서 음수가 된다.

```
TQQQ 40분할: 별% = (15 - 0.75 × T) %
SOXL  40분할: 별% = (20 - T) %
TQQQ 20분할: 별% = (15 - 1.5 × T) %
SOXL  20분할: 별% = (20 - 2 × T) %
```

### 별지점 (★지점)

매수와 매도를 가르는 기준 가격.

```
별지점 = 평단 × (1 + 별%)
LOC 매수가 = 별지점 - 0.01   (매수는 별지점 바로 아래)
LOC 매도가 = 별지점           (매도는 별지점 그대로)
```

### LOC (Limit On Close)

장 마감 시 특정 가격에 도달하면 체결하는 주문 유형. 종가 주문이라고도 함.

### 1회 매수액

그날 투입할 금액의 기준.

```
1회 매수액 = 잔금 / (분할수 - T)
```

### 쿼터매도

별지점에서 보유수량의 1/4을 LOC 매도하는 것. 전반전/후반전 공통.

```
쿼터매도 수량 = floor(보유수량 / 4)   (내림, 소수점 버림)
```

### 모드

| 모드명 | 조건 | 특징 |
|---|---|---|
| 사이클시작 | T = 0 | 큰수매수. 전날종가 +10% LOC |
| 일반-전반전 | 0 < T < 분할수/2 | 1회매수액 절반을 별지점LOC + 절반을 평단LOC |
| 일반-후반전 | 분할수/2 ≤ T < 분할수 | 1회매수액 전체를 별지점LOC (별지점이 평단 아래) |
| 리버스모드 | T > 분할수 - 1 | 별지점 = 5거래일 종가 평균. 매도 우선, 쿼터매수 |

### 큰수매수

별가격·평단과 실제 종가 괴리가 커서 증권사가 LOC 오류를 낼 때 사용하는 우회 방식.
현재가 +10~15% LOC 가격에 그 위 모든 수량을 합산해서 한 번에 건다.

## 4. 모드 상태 전이

```
[새 사이클 시작]
       │
T = 0  ↓
              ┌──── 사이클시작 ────┐
              │   (큰수매수 LOC)    │
              │   T=0 → T≥1 시    │
              └─────────┬──────────┘
    │ 첫 매수 체결 후 T ≥ 1
    ↓
              ┌─── 일반-전반전 ────┐
              │  0 < T < 분할수/2  │
              │  별지점LOC + 평단LOC│
              └─────────┬──────────┘
    │ T ≥ 분할수/2
    ↓
              ┌─── 일반-후반전 ────┐
              │  분할수/2 ≤ T < 분할수│
              │  별지점LOC 전액     │
              └─────────┬──────────┘
    │ T > 분할수 - 1
    ↓
              ┌──── 리버스모드 ────┐
              │  별지점 = 5일 평균  │
              │  첫날 MOC 매도만    │
              │  이후 쿼터매도+매수 │
              └─────────┬──────────┘
    │ 종가 > 평단 × 0.85 (TQQQ)
    │ 종가 > 평단 × 0.80 (SOXL)
    ↓
              ┌─── 일반-전반전 ────┐ (T값 그대로 연결)
              └───────────────────┘

별도 분기:
              일반모드 중 보유수량 = 0
    │
    ↓
              사이클 종료 → 복리/단리 선택 → T=0 리셋
```

## 5. 계산 공식 전체 명세

> 이 공식들을 기반으로 서버 로직이 구현된다. 디자이너는 이 공식에서 어떤 값을 UI에 표시해야 하는지 파악한다.

### 5-1. 매수계획 계산 (일반-전반전)

```
1회 매수액    = 잔금 / (분할수 - T)
별지점        = 평단 × (1 + 별%)
LOC 매수가_별 = 별지점 - 0.01
LOC 매수가_평 = 평단 (그대로)

절반 금액   = 1회 매수액 / 2
별지점 수량 = floor(절반 금액 / LOC 매수가_별)   → 0이면 1주로 올림
평단 수량   = floor(절반 금액 / LOC 매수가_평)   → 0이면 1주로 올림

※ 별지점 1주 + 평단 1주 합계가 1회매수액 초과해도 허용 (방법론 취지)
```

### 5-2. 매수계획 계산 (일반-후반전)

```
1회 매수액    = 잔금 / (분할수 - T)
별지점        = 평단 × (1 + 별%)    ← 별%가 음수이므로 별지점 < 평단
LOC 매수가_별 = 별지점 - 0.01

별지점 수량 = floor(1회 매수액 / LOC 매수가_별)
```

### 5-3. 매수계획 계산 (사이클시작, T=0)

```
큰수 LOC 가격 = 전날 종가 × 1.10   (10% 위)
큰수 수량     = floor(1회 매수액 / 큰수 LOC 가격)
분할 LOC는 큰수 가격 아래로 매 단계 7% 할인씩 쭉 1주씩
```

### 5-4. 매도계획 계산 (전반전/후반전 공통)

```
쿼터매도가    = 별지점              (LOC 매도)
쿼터매도 수량 = floor(보유수량 / 4)

지정가 매도가 = 평단 × 1.15        (TQQQ, 15% 위)
지정가 매도가 = 평단 × 1.20        (SOXL,  20% 위)
지정가 수량   = 보유수량 - 쿼터매도 수량
```

### 5-5. 리버스모드 별지점

```
리버스 별지점 = 직전 5거래일 종가의 평균
LOC 매도가    = 리버스 별지점
LOC 매수가    = 리버스 별지점 - 0.01
```

### 5-6. 리버스모드 매도 수량

```
첫날 매도 수량 = floor(보유수량 / 20)    (40분할 기준)
이후 매도 수량 = floor(직전 보유수량 / 20)  (매일 재계산)
```

### 5-7. 리버스모드 쿼터매수

```
매수 금액 = 잔금 / 4
매수 수량 = floor(매수 금액 / LOC 매수가)
```

### 5-8. T값 갱신 (리버스모드)

```
매도 후: T = T × 0.95
매수 후: T = T + (분할수 - T) × 0.25
```

### 5-9. 리버스모드 종료 조건

```
TQQQ: 종가 > 평단 × 0.85   → 일반모드 복귀
SOXL: 종가 > 평단 × 0.80   → 일반모드 복귀
```

### 5-10. 평단 갱신

```
매수 시: 평단 = (평단_old × 수량_old + 체결가 × 체결수량) / (수량_old + 체결수량)
매도 시: 평단 변동 없음
```

### 5-11. 분할 LOC 가격 (큰 하락 대비 추가 매수라인)

```
별지점/평단 기준 가격에서 매 단계 약 7% 할인 적용, 1주씩
가격_n = 가격_n-1 × (1 - 0.07)   ← 방법론상 가이드, 정확한 배율은 유연
```

## 6. 리버스모드 상세 플로우

리버스모드는 손절 구간이다. 매도 우선, 매수는 쿼터만.

### 진입 조건

- 40분할: T > 39
- 20분할: T > 19

### 첫날

- **매도만**: floor(보유수량 / 20) 개를 MOC(Market On Close, 무조건 체결) 매도
- **매수 없음**
- T 갱신: `T = T × 0.95`

### 둘째날 이후 (매일 반복)

1. 별지점 계산: 직전 5거래일 종가 평균
2. 종가 기준 판단:
   - 종가 ≥ 리버스 별지점 → 전날에 LOC 매도 체결됨
   - 종가 < 리버스 별지점 → 전날에 LOC 매수 체결됨
3. 매도: floor(직전 보유수량 / 20) 개를 리버스 별지점 LOC
4. 매수: 잔금 / 4 금액으로 리버스 별지점 - 0.01 LOC

### 종료 판단

매일 장 마감 후 체결 입력 시:

```
if 종가 > 평단 × 0.85 (TQQQ) or 평단 × 0.80 (SOXL):
    → 리버스모드 종료, 일반모드로 복귀
    → T값은 리버스모드에서 쓰던 값 그대로 연결
```

## 7. 데이터 모델 (PostgreSQL 스키마)

스택: **NestJS + TypeORM(또는 Prisma) + PostgreSQL** (Mac mini 자체 호스팅)

### `users`

```sql
id                   uuid        PK  DEFAULT gen_random_uuid()
email                text        UNIQUE NOT NULL
password_hash        text        NOT NULL
refresh_token_hash   text        -- 리프레시 토큰 해시 (로그아웃 시 null)
created_at           timestamptz DEFAULT now()
```

### `strategies`

전략 인스턴스. 사용자당 TQQQ, SOXL 각각 1개 (또는 복수 가능).

```sql
id          uuid     PK  DEFAULT gen_random_uuid()
user_id     uuid     FK → users.id  NOT NULL
ticker      text     NOT NULL  -- 'TQQQ' | 'SOXL'
division    int      NOT NULL  -- 분할수. 20 or 40
principal   numeric  NOT NULL  -- 원금 ($)
cycle_no    int      NOT NULL DEFAULT 1
created_at  timestamptz DEFAULT now()
```

### `state`

전략별 현재 상태. 1 strategy = 1 row (upsert).

```sql
strategy_id   uuid     PK, FK → strategies.id
mode          text     NOT NULL  -- 'cycle_start' | 'first_half' | 'second_half' | 'reverse'
t_value       numeric  NOT NULL  -- T값 (소수점 무제한)
avg_price     numeric            -- 평단 ($). 사이클시작(T=0)이면 null
quantity      numeric  NOT NULL DEFAULT 0  -- 보유수량 (주)
cash          numeric  NOT NULL  -- 잔금 ($)
last_close    numeric            -- 마지막 종가
recent_closes numeric[]          -- 최근 5거래일 종가 (리버스 별지점용)
updated_at    timestamptz DEFAULT now()
```

### `daily_plans`

날짜별 매수매도표. 계산 결과를 저장해서 UI에서 빠르게 조회.

```sql
id            uuid     PK  DEFAULT gen_random_uuid()
strategy_id   uuid     FK → strategies.id  NOT NULL
plan_date     date     NOT NULL
mode          text
t_value       numeric
avg_price     numeric
cash          numeric
once_amount   numeric       -- 1회 매수액
star_price    numeric       -- 별지점
buy_rows      jsonb         -- [{label, loc_price, qty}, ...]
sell_rows     jsonb         -- [{label, price, qty}, ...]
created_at    timestamptz DEFAULT now()
UNIQUE (strategy_id, plan_date)
```

### `executions` (체결 히스토리)

```sql
id            uuid     PK  DEFAULT gen_random_uuid()
strategy_id   uuid     FK → strategies.id  NOT NULL
exec_date     date     NOT NULL
exec_type     text     NOT NULL
              -- 'buy_full' | 'buy_half_star' | 'buy_half_avg'
              -- | 'sell_quarter' | 'sell_fixed' | 'sell_moc' | 'no_exec'
exec_price    numeric
exec_qty      numeric
exec_amount   numeric       -- 체결금액
t_before      numeric
t_after       numeric
avg_before    numeric
avg_after     numeric
cash_before   numeric
cash_after    numeric
note          text          -- 큰수매수, 매도거부 등 메모
created_at    timestamptz DEFAULT now()
```

### `prices` (시세 캐시)

```sql
ticker        text
price_date    date
close_price   numeric
fetched_at    timestamptz
PRIMARY KEY (ticker, price_date)
```

## 7-B. REST API 엔드포인트 일람

프론트가 백엔드(NestJS)와 통신하는 엔드포인트. 모든 요청에 `Authorization: Bearer <access_token>` 필요 (Auth 제외).

### Auth

```
POST   /auth/login         body: {email, password}          → {access_token, refresh_token}
POST   /auth/refresh       body: {refresh_token}            → {access_token, refresh_token}
POST   /auth/logout        header: Bearer token             → 204
POST   /auth/signup        body: {email, password}          → {user_id}
```

### Strategies

```
GET    /strategies       → [{id, ticker, division, principal, cycle_no}]
POST   /strategies         body: {ticker, division, principal} → {id}
PATCH  /strategies/:id     body: {principal?, cycle_no?}    → {id}
DELETE /strategies/:id  → 204
```

### State

```
GET    /strategies/:id/state            → 현재 state 전체
```

### Plans

```
GET    /strategies/:id/plans/today      → 오늘 daily_plan (없으면 계산 후 반환)
GET    /strategies/:id/plans?date=YYYY-MM-DD                → 특정 날 plan
```

### Executions

```
GET    /strategies/:id/executions       → 체결 히스토리 목록 (최신순)
POST   /strategies/:id/executions  body: [{exec_type, exec_price, exec_qty, exec_date, note}]
    → state 갱신 + 내일 plan 재계산 → {new_state, tomorrow_plan}
```

### Prices

```
GET    /prices/:ticker/latest           → {ticker, price_date, close_price}
POST   /prices/:ticker/refresh         → 외부 API 수동 fetch 트리거 → {close_price}
```

## 12. 시세 API

### 아키텍처 원칙

**클라이언트는 외부 API를 직접 호출하지 않는다.** NestJS 백엔드가 외부 시세 API의 프록시+캐시 역할을 한다.

```
프론트 → GET /prices/:ticker/latest → PricesModule → prices 테이블 캐시
                ↓ 캐시 없거나 만료 시
           외부 API (Yahoo Finance 등) fetch
```

### 자동 fetch (Cron)

NestJS `@nestjs/schedule` 모듈로 미국 장 마감 후 자동 실행.

```
@Cron('30 16 * * 1-5', { timeZone: 'America/New_York' })
// 평일 ET 16:30 (서머타임 자동 대응)
→ TQQQ, SOXL 종가 fetch → prices 테이블 upsert
```

### 추천 소스 (무료)

- **Yahoo Finance Unofficial** (axios로 직접 파싱 or `yahoo-finance2` npm)
- **Alpha Vantage** 무료 tier (분당 5회, 하루 25회 제한)

### 수동 입력 Fallback

사용자가 "시세 새로고침" 버튼 클릭 → `POST /prices/:ticker/refresh` 수동 트리거.
자동 fetch가 실패한 경우 UI에서 직접 종가 입력 가능.

### 5거래일 종가 보관

리버스모드 별지점 계산용. `state.recent_closes` 배열에 최신 5개 유지.
매일 `POST /strategies/:id/executions` 저장 시 백엔드가 자동 append + 6번째 제거.

## 13. 엣지 케이스

### 13-1. 큰수매수

**발생 조건**: 별가격(또는 평단)과 현재 종가의 괴리가 10~20% 이상 → 증권사가 LOC 오류 반환

**UI 처리**:
- 계획 표에 큰수매수 경고 배너 표시:
  > "별지점($83.82)이 종가($62.00)와 35% 차이. 큰수매수 필요 — 종가 +10~15% 위 가격으로 설정하세요."
- 큰수 LOC 가격 자동 제안: `종가 × 1.15` (사용자 조정 가능)
- 해당 가격 위 모든 수량 합산해서 큰수 행에 표시

**원칙**: 별가격 아래에서 무조건 매수가 걸려야 한다. 가격이 몇 % 위냐보다 이 원칙이 우선.

### 13-2. 매도거부

**발생 조건**: 큰 하락 시 별지점 매도가가 현재가와 너무 멀어 증권사가 거부

**UI 처리**:
- 매도점 표에 안내 텍스트:
  > "종가 괴리 과대 시 매도 미체결 가능. 낮은 가격으로 변경 절대 금지."
- 체결 입력 시 "미체결(매도거부)" 구분 선택지 제공

### 13-3. LOC 정수 단위

- 최소 주문 단위: 1주 (정수만)
- 절반 수량 계산 결과가 0이면 자동으로 1주로 올림
- 별지점 1주 + 평단 1주 합계가 1회매수액을 초과해도 허용 (방법론 취지)
- 초과분은 다음날 `잔금 / (분할수 - T)` 재계산으로 자연 흡수

### 13-4. 사이클 종료

```
if 보유수량 == 0:
    → 사이클 종료 화면 표시
    → 사용자: 복리 or 단리 선택
    → 복리: 새 원금 = 잔금 (수익 포함)
    → 단리: 새 원금 = 최초 설정 원금
    → strategies.cycle_no += 1
    → state 초기화 (T=0, 평단=null, 보유=0)
```

### 14-1. 전반적 톤

- 금융 대시보드. 차분하고 신뢰감 있는 톤.

### 14-3. 표 형식 유지 원칙

방법론 원문의 3컬럼 표(구분·가격·수량)를 그대로 유지한다. 사용자의 멘탈모델.
디자이너 임의로 카드형·슬라이더형 등으로 변환 금지. 모바일에서 가로 스크롤은 허용.

```
< 매수점 >
| 구분   | LOC 가격 | 수량 |
|--------|---------|------|
| ★ LOC  | $83.82  |  1  |
| 평단   | $75.10  |  1  |
| 분할③  | $69.84  |  1  |

< 매도점 >
| 구분      | 가격    | 수량 |
|-----------|---------|------|
| ★% LOC   | $83.83  |  2  |
| 15% 지정  | $86.37  |  6  |
```

### 14-4. 모바일 디자인 필수 가이드라인

**터치 타겟**
- 모든 버튼·탭·아이콘: 최소 **44×44pt** (iOS HIG 기준)
- CTA 버튼 ("어제 체결 입력", "저장"): 높이 **56dp**, 전체 너비

**폰트 크기**
- 본문 (상태값, 표 셀): 최소 **16px** (모바일 가독성)
- 가격·T값 등 숫자: **모노스페이스**, 16~20px
- 섹션 헤더: 14px, 대문자 + 자간 넓게

**간격 / 여백**
- 카드 내부 패딩: 16px
- 카드 간 간격: 12px
- 표 행 높이: 최소 44px

**Safe Area 대응**
- iOS: `env(safe-area-inset-top)`, `env(safe-area-inset-bottom)` 반드시 적용
- Bottom Tab Bar: 하단 safe area 위에 고정
- Bottom Sheet CTA 버튼: 홈 인디케이터 위로 충분한 패딩

**가로 스크롤**
- 매수매도표: 컬럼 수 고정, 컨테이너에 `overflow-x: auto`
- 페이지 전체 가로 스크롤 금지

**Pull-to-Refresh**
- 대시보드 홈: 위로 당기면 시세 새로고침 트리거

## 부록 A. 실제 데이터 예시 (TQQQ 현재 상태)

Claude Design이 UI 목업을 만들 때 이 데이터를 예시로 사용한다.

```
종목: TQQQ
분할: 40분할
원금: $4,800.00
사이클: 1

T값: 4.5
모드: 일반-전반전
평단: $75.10
보유: 8주 (실제 8.717169주, UI 표시는 정수)
잔금: $4,145.31
1회 매수액: $4,145.31 / (40 - 4.5) = $116.77

별%: (15 - 0.75 × 4.5) = 11.625%
별지점: $75.10 × 1.11625 = $83.83
LOC 매수가: $83.82
LOC 매도가: $83.83
15% 지정가: $86.37

쿼터매도 수량: floor(8.717169 / 4) = 2주
지정가 매도 수량: 8 - 2 = 6주
```
